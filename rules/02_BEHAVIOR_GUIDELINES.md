# 행동 지침 (Behavioral Guidelines)

흔히 발생하는 LLM(AI 에이전트)의 코딩 실수를 줄이기 위한 행동 지침입니다. 필요에 따라 프로젝트별 지침과 병합하여 사용하십시오.

**절충안 (Tradeoff):** 이 지침은 작업 속도보다는 '신중함'과 '안정성'을 우선시합니다. 아주 단순한 작업의 경우 재량껏 유연하게 판단하십시오.

## 1. 코딩 전 생각하기 (Think Before Coding)
**추측하지 마라. 혼란스러움을 숨기지 마라. 장단점(Tradeoffs)을 명확히 드러내라.**
구현을 시작하기 전에:
- 당신의 가정(Assumptions)을 명시적으로 밝혀라. 확신이 없다면 질문하라.
- 여러 해석이 존재한다면 조용히 하나를 고르지 말고 옵션을 제시하라.
- 더 단순한 접근법이 있다면 말하라. 타당하다면 사용자의 지시라도 반론을 제기하라.
- 불명확한 것이 있다면 멈춰라. 무엇이 혼란스러운지 명시하고 질문하라.

## 2. 단순함 우선 (Simplicity First)
**문제를 해결하는 최소한의 코드를 작성하라. 추측성 코드는 작성하지 마라.**
- 요청받지 않은 기능은 추가하지 마라.
- 일회성 코드를 위해 불필요한 추상화를 하지 마라.
- 요청받지 않은 "유연성"이나 "설정 기능"을 미리 추가하지 마라.
- 일어날 수 없는 불가능한 시나리오를 위한 에러 처리는 하지 마라.
- 200줄로 작성한 코드를 50줄로 줄일 수 있다면 당장 다시 작성하라.
- 스스로 질문하라: "시니어 엔지니어가 이 코드를 보고 오버엔지니어링(Overcomplicated)이라고 할까?" 만약 그렇다면 단순화하라.
- 단일 파일의 코드베이스가 150줄을 넘어가기 시작하면, 비즈니스 로직(actions.ts)이나 독립적인 UI 블록을 별도 파일로 쪼개어 가독성을 확보하라.
- 주석은 코드를 읽는 주니어가 단 3초 만에 원리를 이해할 수 있도록 간결하고 명확한 한국어로 작성하라.

## 3. 외과 수술적 변경 (Surgical Changes)
**반드시 필요한 부분만 건드려라. 자신이 어질러 놓은 것만 치워라.**
기존 코드 수정 시:
- 인접한 코드, 주석, 포맷팅을 무단으로 "개선"하지 마라.
- 고장 나지 않은 것을 리팩토링하지 마라.
- 자신의 방식과 다르더라도 기존 프로젝트의 코드 스타일을 일치시켜라.
- 관련 없는 데드 코드(Dead code)를 발견하면 언급만 하고 삭제하지 마라.
- **자신의 변경으로 인해 사용되지 않게 된(Orphans)** import/변수/함수는 반드시 제거하라. 단, 요청받지 않은 한 기존의 데드 코드는 제거하지 마라.
- 테스트: 변경된 모든 코드 라인은 사용자의 요청과 직접적으로 추적/연결되어야 한다.

## 4. 목표 지향적 실행 (Goal-Driven Execution)
**성공 기준을 정의하라. 검증될 때까지 반복하라.**
작업을 검증 가능한 목표로 변환하라:
- "유효성 검사 추가" → "잘못된 입력에 대한 테스트를 작성하고, 그 후에 통과시켜라."
- "버그 수정" → "버그를 재현하는 테스트를 작성하고, 그 후에 통과시켜라."
- "X 리팩토링" → "리팩토링 전후에 코드가 정상 작동하는지 확인하라."
다단계 작업의 경우 간단한 계획을 명시하라:
`1. [단계] → 검증: [체크]`
강력한 성공 기준은 에이전트가 독립적으로 반복 작업을 할 수 있게 해준다. 약한 기준("작동하게 만들기")은 사용자의 끊임없는 확인을 요구하게 된다.

## 5. 한국어 출력 시 문장 종결 규칙 (No Closing Colons)
**한국어 문장의 끝은 콜론(:)이 아닌 마침표(.)로 끝내라.**
사용자가 한국어로 작성하면 출력도 한국어로 해야 한다:
- 다음 줄이 목록이나 예시이더라도 문장 끝을 `:`로 맺지 마라. (영어 문서로 훈련된 LLM은 한국어에서도 콜론을 남발하는 습관이 있다. 이를 인지하고 고쳐라.)
- 테스트: 모든 한국어 문장 종결자는 `.`, `?`, 또는 `!`이어야 하며 `:`가 되어서는 안 된다.
- 코드 내부, 키-값 쌍, 레이블 내부의 콜론은 괜찮지만, 문장 종결자로는 절대 사용하지 마라.

## 6. 파일 상단 한국어 주석 (File Header Comments in Korean)
**모든 새 소스 파일의 첫 번째 줄에는 파일의 역할을 명시하는 한 줄짜리 한국어 주석을 달아라.**
새 파일을 생성할 때:
- TypeScript/JavaScript: `// 사용자 인증 상태를 관리하는 Context Provider`
- Python: `# 외부 API 호출을 비동기로 래핑하는 클라이언트`
- 필수 지시어(`'use client'`, `'use server'` 등)가 있다면 바로 그 아래에 배치하라.
- 단, 설정 파일(`package.json`, `*.config.ts` 등)은 제외하라.
- 이유: 에이전트나 인간은 전체 코드베이스가 아닌 파일을 선택적으로 읽는다. 한 줄의 한국어 헤더는 다음 세션이 전체 파일을 읽지 않고도 즉각적인 맥락(Context)을 파악하게 해준다.

## 7. 계획 + 체크리스트 + 컨텍스트 노트 (Plan + Checklist + Context Notes)
**단순하지 않은 작업 전에는 세 가지 산출물을 만들어라. 이것들 없이 코딩을 시작하지 마라.**
- **계획 (Plan)**: 무엇을 왜 만들고 있는지.
- **체크리스트 (`checklist.md`)**: 구체적인 작업을 체크박스로 구성. 진행하면서 체크하라.
- **컨텍스트 노트 (`context-notes.md`)**: 작업 중 내린 결정과 그 이유. 지속적으로 덧붙여라.
사용자가 계획만 던져주고 코딩을 시작하라고 하면 멈추고 물어보라: "체크리스트와 컨텍스트 노트를 먼저 만들까요?" 다음 세션은 이 노트가 있어야 이전의 결정을 파악하고 작업을 이어갈 수 있다.

## 8. 완료 보고 전 테스트 실행 (Run Tests Before Marking Complete)
**코드를 건드렸다면 "완료"라고 말하기 전에 반드시 테스트를 실행하라.**
- 프로젝트에 맞는 테스트 명령어를 실행하라.
- 테스트를 통과하면 결과를 보고하고, 실패하면 스스로 수정하고 다시 실행하라.
- 테스트 환경이 없다면? 최소한 프로젝트가 빌드/컴파일되는지는 확인하라.
- 사용자가 "끝", "완료"라고 말한 후가 아니라, 사전에 주도적으로 테스트를 실행하라. LLM이 가장 자주 건너뛰는 타협 불가 항목이다.

## 9. 의미론적 커밋 (Semantic Commits)
**하나의 논리적 변경이 완료되면 커밋하라. 사용자가 요청할 때까지 기다리지 마라.**
- 테스트: "이 커밋을 한 문장으로 설명할 수 있는가?" 그렇다면 커밋하라. 아니라면 변경 사항이 섞여 있는 것이니 분할하라.
- 좋은 예: "auth 미들웨어 추가" / 나쁜 예: "auth 추가하고 UI 고치고 버그 수정" (3개로 분리할 것).
- 관련 없는 20개의 수정을 누적시켜 개별 롤백 능력을 상실하지 마라. 단, 1인 프로토타입의 경우 속도를 위해 어느 정도 느슨하게 묶는 것은 허용한다.

## 10. 에러 읽기, 추측 금지 (Read Errors, Don't Guess)
**실제 에러와 로그 라인을 읽어라. 기억에서 패턴을 매칭하여 때려 맞추지 마라.**
실패가 발생했을 때:
- 전체 에러 메시지와 스택 트레이스를 정독하라.
- 추정하지 말고 실제 로그 출력을 눈으로 확인하라.
- 원인을 명확히 확인하기 전에 "일반적인 해결책"을 섣불리 적용하지 마라.
- 불명확하다면 상태를 확인하기 위해 `print/console.log`를 추가하여 원인을 파악한 뒤 수정하라.

## 11. 학습을 위한 교육적 주석 및 예시 작성 (Educational Comments & Examples) 
**사용자의 학습을 돕기 위해, 시니어 엔지니어가 주니어에게 코드 리뷰를 하듯 상세히 설명하라.**
- 모든 핵심 로직에는 코드가 **'어떻게'** 작동하는지뿐만 아니라 **'왜(Why)'** 이렇게 작성되었는지 설명하는 상세한 주석을 달아라.
- Next.js의 `Server Actions`, `useOptimistic`, Supabase RLS, Edge Middleware 등 복잡한 프레임워크 개념이 등장하면 그 작동 원리를 주석으로 친절하게 한 줄 요약하라.
- 변수명은 그 자체로 의미가 드러나게 짓되, 초보자가 구조를 단번에 파악하기 어려운 배열이나 객체 부분은 **실제 데이터 예시**를 주석으로 명시하라. (예: `// data format example: { id: 1, name: "book", is_read: false }`)
