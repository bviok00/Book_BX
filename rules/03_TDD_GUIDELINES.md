### 📘 AI-Native TDD Master Blueprint (One-Turn Execution)

이 프로젝트의 모든 기능 구현 및 리팩토링은 비용(토큰) 최소화와 코드 무결성을 동시에 달성하기 위해 아래의 'AI 네이티브 TDD 워크플로우'를 절대적으로 준수해야 합니다.

#### 📌 1. 핵심 원칙 (Core Principles)

- **One-Turn TDD (초압축 사이클):** AI는 텍스트를 출력하기 전 내부 연산으로 Red(실패) 상태를 이미 시뮬레이션할 수 있습니다. 따라서 사용자의 승인을 기다리며 대화를 끊지 마십시오. **단 한 번의 답변(Turn) 내에서 [테스트 코드 작성]과 [통과하는 구현 코드 작성]을 연속으로 모두 출력**하여 즉시 Green 상태를 만들어내야 합니다.
    
- **테스트 타겟팅 (No E2E):** 브라우저의 비동기 렌더링을 직접 제어해야 하는 E2E 테스트(Playwright 등)는 AI의 환각(Hallucination) 루프를 유발하므로 이 단계에서 **절대 작성하지 마십시오.**
    
- **다계층 검증:** 오직 아래 2가지에 집중하며, 핵심 비즈니스 로직의 커버리지는 80% 이상을 확보하십시오.
    
    1. **Unit Test:** UI 컴포넌트 렌더링, 개별 훅(Hooks), 유틸 함수 검증.
        
    2. **Integration Test:** 5대 ZONE 구조 간의 데이터 흐름(actions.ts ↔ UI 컴포넌트) 및 API 모킹 검증.
        

#### 🔄 2. 실행 프로세스 (The One-Turn Execution)

새로운 태스크를 부여받으면 반드시 다음 순서대로 단일 텍스트 블록 안에 응답을 구성하십시오.

1. **[Red] 테스트 코드 먼저 출력:** 요구사항을 검증하는 실패할 테스트 코드(`*.test.ts/tsx`)를 제시합니다.
    
2. **[Green] 즉시 구현 코드 출력:** 연달아 해당 테스트를 100% 통과할 수 있는 프로덕션 코드를 제시합니다.
    
3. **[Refactor] 클린 아키텍처 점검:** 작성된 코드가 5대 ZONE 구조(화면과 로직의 분리)를 어기지 않았는지 스스로 점검하고 최적화된 상태로 제출합니다.
    

#### 🛠 3. 에러 대응 및 뇌절 금지 (Strict Anti-Yapping Rule)

작업 후 테스트가 실패하여 디버깅을 해야 할 경우, 장황한 사과나 원인 설명("이것은 비동기 렌더링 지연 문제이며...")을 **절대 금지**합니다. 오직 아래의 초간결 포맷으로만 즉시 답변하십시오.

- **[❌ FAIL]** `실패한 테스트명 또는 에러 로그 한 줄`
    
- **[🔧 FIX]** `문제를 해결한 핵심 코드 스니펫`
